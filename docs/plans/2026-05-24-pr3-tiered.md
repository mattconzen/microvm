# PR3 — Tiered snapshot mode (S3 Files + AgentCore session storage)

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Land the `tiered` snapshot mode end-to-end. After this PR, `microvm login --snapshot-mode tiered --s3-files-access-point-arn …` works, and snapshot/resume copy a per-sandbox prefix of an S3 Files mount via server-side `aws s3 cp --recursive`. A new `microvm sbx checkpoint <id>` command rsyncs from the fast cache tier into the snapshottable workspace tier. Existing `none`/`s3`/`efs` modes are unchanged.

**Architecture:**
- Two filesystems are visible inside the runtime:
  - **Tier 1 (cache):** `/var/microvm/cache/<sandbox_id>/` — ordinary container storage; AgentCore "managed session storage" preserves it for the session lifetime and discards it on eviction. Real POSIX (rename, locks, xattrs). 1 GB-ish cap; not snapshotted.
  - **Tier 2 (workspace):** `/workspace/<sandbox_id>/` — AgentCore S3 Files access-point mount. Each sandbox is isolated by prefix. Mountpoint POSIX gaps (no rename, no random writes) but durable, snapshottable in O(seconds) via server-side `CopyObject`.
- Snapshot = `aws s3 cp --recursive s3://<bucket>/sessions/<sb>/ s3://<bucket>/snapshots/<snap>/` (server-side, no data transfer through the runtime). Resume is the reverse. We run these from inside the shellagent for symmetry with the EFS path; a future PR can move them CLI-side for tighter IAM (see "Open questions").
- New `microvm sbx checkpoint <id>` runs `rsync -a` from `<cache>/promote/` to `<workspace>/cache-promoted/`. Users opt-in by writing into the `promote/` subdirectory anything they want included in the next snapshot.
- Shellagent default CWD for sbx-exec commands is `/workspace/<sandbox_id>/` when mode=tiered. Interactive shells get `MICROVM_WORKSPACE` and `MICROVM_CACHE` env vars; the user types `cd $MICROVM_WORKSPACE` themselves (shell-init protocol upgrade is deferred — see "Open questions").

**Tech stack:** Go 1.22, Python 3.12, AWS SDK v2 (Go), boto3-free Python (shells out to `aws s3 cp`), bash + aws CLI for setup, pytest, testify.

```go
package aws

import (
    "fmt"
    "regexp"
    "strings"

    "github.com/mattconzen/microvm/backend"
)

const (
    tieredDefaultMountPath = "/workspace"
    tieredCacheRoot        = "/var/microvm/cache"
)

// Resource segment of an S3 access-point ARN, e.g. accesspoint/microvm-files.
var s3AccessPointResourceRe = regexp.MustCompile(`^accesspoint/[A-Za-z0-9._-]+$`)

type tieredProvisioner struct {
    accessPointArn string
    bucket         string
    mountPath      string
}

func newTieredProvisioner(opts backend.LoginOpts) (tieredProvisioner, error) {
    arn := strings.TrimSpace(opts.S3FilesAccessPointArn)
    if arn == "" {
        return tieredProvisioner{}, fmt.Errorf("snapshot mode %q requires --s3-files-access-point-arn", "tiered")
    }
    if !isValidS3AccessPointArn(arn) {
        return tieredProvisioner{}, fmt.Errorf("invalid S3 access point ARN %q", arn)
    }
    bucket := strings.TrimSpace(opts.S3FilesBucket)
    if bucket == "" {
        return tieredProvisioner{}, fmt.Errorf("snapshot mode %q requires --s3-files-bucket", "tiered")
    }
    if !isValidBucketName(bucket) {
        return tieredProvisioner{}, fmt.Errorf("invalid S3 bucket name %q", bucket)
    }
    mp := strings.TrimSpace(opts.S3FilesMountPath)
    if mp == "" {
        mp = tieredDefaultMountPath
    }
    if !strings.HasPrefix(mp, "/") {
        return tieredProvisioner{}, fmt.Errorf("S3 Files mount path %q must be absolute", mp)
    }
    return tieredProvisioner{accessPointArn: arn, bucket: bucket, mountPath: mp}, nil
}

func (tieredProvisioner) Mode() string { return "tiered" }

func (p tieredProvisioner) ValidateLoginOpts(opts backend.LoginOpts) error {
    if strings.TrimSpace(opts.S3FilesAccessPointArn) == "" {
        return fmt.Errorf("snapshot mode %q requires --s3-files-access-point-arn", "tiered")
    }
    if strings.TrimSpace(opts.S3FilesBucket) == "" {
        return fmt.Errorf("snapshot mode %q requires --s3-files-bucket", "tiered")
    }
    return nil
}

func (p tieredProvisioner) EnvOverrides() map[string]string {
    return map[string]string{
        "MICROVM_SNAPSHOT_MODE":      "tiered",
        "MICROVM_S3FILES_BUCKET":     p.bucket,
        "MICROVM_S3FILES_MOUNT_PATH": p.mountPath,
        "MICROVM_CACHE_ROOT":         tieredCacheRoot,
    }
}

func isValidS3AccessPointArn(arn string) bool {
    parts := strings.Split(arn, ":")
    if len(parts) != 6 {
        return false
    }
    if parts[0] != "arn" || parts[1] != "aws" || parts[2] != "s3" {
        return false
    }
    return s3AccessPointResourceRe.MatchString(parts[5])
}
```

**Step 3 — Implement `TieredSnapshotter`** in `snapshotter.py`. It uses the `_safe_id` helper PR2 already added. The shape mirrors `S3Snapshotter` but the body uses `aws s3 cp --recursive` for server-side prefix copy. Wrap the AWS CLI call in a `_aws_cp_recursive(src, dst)` method so tests can override it.

Pseudocode for the class:

```
class TieredSnapshotter(Snapshotter):
    mode = "tiered"

    __init__(self, bucket, mount="/workspace"):
        self.bucket = bucket
        self.mount  = mount

    snapshot(self, snap_id, name, sandbox_id=""):
        validate ids via _safe_id; on failure return error envelope
        src = f"s3://{bucket}/sessions/{sandbox_id}/"
        dst = f"s3://{bucket}/snapshots/{snap_id}/"
        call _aws_cp_recursive(src, dst); on CalledProcessError return error
        return {alias, name, locator=json({workspace_prefix:src, snapshot_prefix:dst})}

    resume(self, locator, alias, sandbox_id=""):
        validate sandbox_id, locator, snapshot_prefix
        dst = f"s3://{bucket}/sessions/{sandbox_id}/"
        call _aws_cp_recursive(snapshot_prefix, dst); on CalledProcessError return error
        return {alias}

    _aws_cp_recursive(self, src, dst):
        run ["aws", "s3", "cp", "--recursive", src, dst] with check=True, capture_output=True
```

The implementation pattern is identical to PR1's S3Snapshotter — same error envelope shape, same `_safe_id` reuse, same subclass-friendly wrapper method.

import (

func newCheckpointCmd(ctx context.Context, app *App, g *GlobalFlags) *cobra.Command {
    cmd := &cobra.Command{
        Use:   "checkpoint <id>",
        Short: "Promote tier-1 cache (promote/) into tier-2 workspace (cache-promoted/)",
        Long: `In tiered mode, the cache tier (/var/microvm/cache/<sandbox_id>/) is fast

        RunE: func(cmd *cobra.Command, args []string) error {

            if err != nil {
                return err
            }
            sbApi := backend.Sandbox{ID: sb.ID, Provider: sb.Provider, SessionID: sb.SessionID, Mode: sb.Mode}
            if err := b.Checkpoint(ctx, sbApi); err != nil {
                return err
            }
            fmt.Fprintf(cmd.OutOrStdout(), "checkpointed %s\n", sb.ID)
            return nil
        },
    }
    return cmd
}

### Working with the cache tier (tiered mode)

In tiered mode, two paths are visible inside the sandbox:

- `/workspace/<sandbox_id>/` (default cwd for `microvm sbx exec`) — durable,
  snapshotted. S3-backed, so no rename, no random writes, no SQLite-WAL.
- `/var/microvm/cache/<sandbox_id>/` — fast POSIX scratch. AgentCore-managed,
  preserved for the session lifetime, lost on eviction.

Tools that need real POSIX (`git`, `npm install`, `pip install`, SQLite) run
faster from the cache. To include cache artifacts in the next snapshot, place
them under `cache/<sandbox_id>/promote/` and run `microvm sbx checkpoint <id>`
to rsync them into the workspace.

## Open questions

1. **CLI-side snapshot routing** — issue #3's design recommends doing the S3 `CopyObject` from the user's CLI rather than the shellagent so the runtime IAM can be denied read access to the `snapshots/*` prefix (defense in depth). PR3 v1 keeps the shellagent-side approach for symmetry with PR1/PR2; v2 can move it CLI-side. Track as a follow-up.
2. **Interactive-shell default cwd** — PR3 only changes the cwd for `microvm sbx exec`. The interactive shell (websocket-based) doesn't know `sandbox_id` at startup; users `cd $MICROVM_WORKSPACE` manually. A future PR can either pipe `sandbox_id` through the websocket handshake or have the shellagent stamp `BEDROCK_AGENTCORE_SESSION_ID`-keyed env vars at startup and have the CLI map sandbox→session.
3. **S3 Files API stability** — at plan-write time the `aws s3files` CLI surface is new. The setup script may need touch-up at implementation time to match the actual current API; budget a small amount of debugging time.
4. **VPC reuse vs separate VPC** — if `setup_efs.sh` already ran, `setup_tiered.sh` should reuse the existing tagged VPC + subnets. We do that by default. If a user runs the scripts in opposite order or has stale state, they may end up with two VPCs. Teardown handles both. Worth a brief README mention.
